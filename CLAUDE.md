# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Restore NuGet packages
dotnet restore

# Build
dotnet build

# Run (Debug)
dotnet run --project YtDownloader --framework net8.0-windows

# Or open YtDownloader.sln in Visual Studio 2022 and press F5
```

There is no test project in this solution and no lint config; `dotnet build` is the only verification step available.

Target framework is `net8.0-windows` (WPF, Windows-only — `UseWPF=true`). The app is a single project (`YtDownloader/YtDownloader.csproj`) referenced by `YtDownloader.sln`.

## Architecture

WPF desktop app in C#/.NET 8 using MVVM, wrapping the external `yt-dlp` and `ffmpeg` binaries directly via `Process` (no wrapper NuGet packages — `YoutubeDLSharp` was removed as an unused, critically-vulnerable dependency).

**DI wiring** happens in `App.xaml.cs` (`ConfigureServices`): all `Services/*` are singletons, all `ViewModels/*` are transient, resolved through `App.Services` (a static `IServiceProvider`). `MainViewModel` resolves its child view models (`DownloadViewModel`, `HistoryViewModel`, `SettingsViewModel`) itself via `App.Services.GetRequiredService<T>()` rather than through constructor injection, since WPF instantiates `MainWindow`/`MainViewModel` directly. **`OnStartup` synchronously loads `ISettingsService` before building any window/ViewModel** (`.LoadAsync().GetAwaiter().GetResult()`) — `DownloadQueueService` sizes its concurrency `SemaphoreSlim` from settings at construction time, and `SettingsViewModel` snapshots settings into bindable properties at construction time, both of which happen the moment `MainViewModel`'s constructor resolves them. If settings load moves back to being async/deferred, both of those silently revert to defaults and a save can clobber the persisted file — keep the synchronous load if you touch this.

**Navigation** is not a `Frame`/router — `MainViewModel.CurrentView` is an `ObservableObject` swapped between the three child view models, driven by `NavigateTo(string view)` and `ShowDownload`/`ShowHistory`/`ShowSettings` booleans bound in `MainWindow.xaml`.

**CommunityToolkit `[RelayCommand(CanExecute=...)]` does not auto-requery.** Any property a `CanExecute` predicate reads needs `[NotifyCanExecuteChangedFor(nameof(TheCommand))]` on it, or the command's enabled state freezes at whatever it evaluated to at construction (see `DownloadViewModel`'s `CurrentVideo`/`IsFetching` — both notify `AddToQueueCommand` and `FetchInfoCommand` for this reason). Easy to reintroduce this bug when adding a new command with a `CanExecute`.

**`Run.Text` bindings default to `TwoWay`** (unlike `TextBlock.Text`, which is `OneWay`), so binding a `<Run Text="{Binding SomeReadOnlyProperty}">` to a get-only property throws a binding error at runtime. Always add `Mode=OneWay` explicitly on `Run.Text` bindings.

**yt-dlp/ffmpeg lifecycle** (`Services/YtDlpService.cs`): on startup `MainViewModel.InitializeAsync()` calls `EnsureInstalledAsync` which downloads `yt-dlp.exe` from the GitHub releases API into `%LocalAppData%\YtDownloader\bin\` if missing (SHA256-verified against the `SHA2-256SUMS` release asset — see below), then optionally `UpdateAsync` if `AppSettings.AutoUpdateYtDlp` is set (failures here are non-fatal — swallowed), then `EnsureFfmpegInstalledAsync` which downloads a static Windows build (`ffmpeg.exe`/`ffprobe.exe`) from the `BtbN/FFmpeg-Builds` GitHub releases into the same `bin\` dir if missing — yt-dlp does **not** bundle ffmpeg itself, and without it audio extraction, format merging, and metadata/thumbnail embedding all fail. Unlike yt-dlp, ffmpeg is **not** integrity-checked — BtbN's releases don't publish checksums. `BuildDownloadArgs` always passes `--ffmpeg-location` pointing at that dir (or `AppSettings.FfmpegPath` if the user overrode it). `DownloadAsync` checks the process exit code and throws (surfacing yt-dlp's `ERROR` lines) on non-zero exit rather than reporting the item as completed. `FetchInfoAsync` calls `--dump-json` and parses the result with `Newtonsoft.Json.Linq` into `VideoInfo`/`VideoFormat`. `BuildFormatSelector` in `DownloadViewModel` (not `YtDlpService`) decides whether to append `+bestaudio/best` to a chosen format ID — only for video-only formats; appending it to an already-muxed/progressive format makes yt-dlp attempt an invalid extra audio merge.

**yt-dlp checksum verification**: `DownloadBinaryAsync` verifies the downloaded `yt-dlp.exe` bytes against yt-dlp's `SHA2-256SUMS` release asset *before* writing anything to disk (`VerifyYtDlpChecksumAsync`/`FindChecksumsAsset`/`TryParseSha256`). Behavior is intentionally asymmetric: a **confirmed hash mismatch throws** (fail-closed — nothing gets written, the error propagates to `MainViewModel`'s startup error state), but an **unparseable/missing checksums asset only warns and proceeds** (fail-open — so a GitHub-side asset rename can't brick installs). Don't make the fail-open case throw too; that was a deliberate design choice.

**Reading yt-dlp's stdout/stderr is unusually fragile — read this before touching `RunAsync`.** Three non-obvious things had to be true simultaneously to get live progress updates working, discovered by empirically bisecting yt-dlp's CLI flags against an isolated test harness (not by reading docs — none of this is documented anywhere):
1. **`PYTHONUNBUFFERED=1` must be set on the child process's environment** (`RunAsync` sets this via `psi.EnvironmentVariables`). yt-dlp is a frozen Python program; when its stdout isn't a real console (i.e. redirected to a pipe, as `Process` does), Python block-buffers instead of line-buffering, so without this, output arrives in large delayed chunks instead of per-line.
2. **Never pass `--print "after_move:...%(filepath)s"`** to capture the final output path, even though it's yt-dlp's officially documented mechanism for this. Empirically confirmed (via an isolated bisection harness, not guesswork) that this flag alone makes yt-dlp buffer *all* of its stdout until the process is about to exit — every other flag combination was tested and ruled out first. Instead, `DownloadAsync` parses the ordinary `[download] Destination:`, `[ExtractAudio] Destination:`, and `[Merger] Merging formats into "..."` lines (`DownloadDestinationRegex`/`ExtractAudioDestinationRegex`/`MergerDestinationRegex`) into a local `capturedPath`, keeping "last one wins" as the final output path once the process exits successfully.
3. **Use manual `StreamReader.ReadLineAsync()` loops (`PumpStreamAsync`), not `Process.OutputDataReceived`/`BeginOutputReadLine()`.** The event-based API was observed to silently stop delivering *any* further events partway through a long-running download — not even the end-of-stream null event — while the process kept running and writing to disk. Root cause unconfirmed (suspected inheritable-pipe-handle interaction with yt-dlp's ffmpeg child process, but not proven); the manual-pump replacement reliably doesn't exhibit it. If you're tempted to "simplify" this back to the event-based API, don't — it's silently broken, not stylistic.

All of `RunAsync`'s internal awaits (`PumpStreamAsync`'s `ReadLineAsync`, `WaitForExitAsync`, `Task.WhenAll`) use `ConfigureAwait(false)` deliberately, so this background I/O work never runs on the captured UI `SynchronizationContext`. `DownloadAsync`'s `onOutput` callback therefore always executes on a genuine ThreadPool thread and marshals every `DownloadItem` property mutation onto the UI thread via `System.Windows.Application.Current.Dispatcher.Invoke(...)` — `DownloadItem` is an `ObservableObject` bound directly to `DownloadView.xaml`, and WPF does not reliably reflect property changes raised off the Dispatcher thread (this was the original "progress bar never moves, then jumps to Completado" bug). `DownloadQueueService.StartProcessingAsync`'s polling loop does **not** use `ConfigureAwait(false)` (unlike `RunAsync`) specifically so it stays on the UI `SynchronizationContext` by default; `ProcessItemAsync`'s own completion/cancel/fail mutations are *also* explicitly `Dispatcher.Invoke`-wrapped as defense-in-depth on top of that. If you add a new place that mutates a `DownloadItem`/`HistoryEntry` property from inside `YtDlpService` or the queue's background paths, wrap it in `Dispatcher.Invoke` too — there's no automatic marshaling anywhere in this codebase.

**Download queue** (`Services/DownloadQueueService.cs`): `StartProcessingAsync` runs a polling loop (started once from `MainViewModel`'s constructor, cancelled via `MainViewModel.Shutdown()`) that pulls the next `Queued` item and dispatches it through a `SemaphoreSlim` gate sized by `AppSettings.MaxConcurrentDownloads` — this is the concurrency control for parallel downloads. Each item runs in `ProcessItemAsync`, which owns a linked `CancellationTokenSource` stored on `DownloadItem.CancellationSource` (used by per-item Cancel). Completed downloads are recorded into `IHistoryService`. All mutations to the underlying `ObservableCollection<DownloadItem>` are marshaled onto the UI thread via `Application.Current.Dispatcher.Invoke`, since the queue is bound directly to a WPF view.

**Persistence**: `SettingsService` and `HistoryService` both serialize plain POCOs (`AppSettings`, `HistoryEntry`) to JSON (`Newtonsoft.Json`) under `%LocalAppData%\YtDownloader\` (`settings.json`, `history.json`). Both fail silently and fall back to defaults/empty on load errors (corrupt file, missing file) — there is no migration or schema versioning.

**History "open file"/"open folder" fallback**: `HistoryViewModel.OpenFile`/`OpenFolder` fall back to opening the configured base output folder (`ISettingsService.Current.OutputFolder`/`AudioOutputFolder`, picked via `entry.AudioOnly`) whenever the stored `OutputPath` is missing or doesn't exist on disk, instead of silently doing nothing — `HistoryViewModel` takes `ISettingsService` as a constructor dependency for this (already DI-registered, no extra wiring needed).

**DownloadItem** (`Models/DownloadItem.cs`) is the observable state machine for a queue entry: `DownloadStatus` enum (`Queued → Fetching/Downloading → Converting → Completed/Failed/Cancelled`) drives derived UI-bound properties (`StatusLabel`, `IsActive`, `IsFinished`, `CanCancel`) recomputed via `OnStatusChanged`/`OnProgressChanged` partial methods (CommunityToolkit.Mvvm source-generated `[ObservableProperty]` pattern used throughout `Models`/`ViewModels`).

**Custom window chrome**: `MainWindow.xaml.cs` implements manual titlebar drag/minimize/maximize/close (`WindowStyle="None"` + `AllowsTransparency="True"`, no default Windows border), so window-chrome logic lives in the view's code-behind rather than a behavior or view model. Because native `Maximize` breaks under `AllowsTransparency` (covers the taskbar, and on multi-monitor setups can size against the wrong monitor), `MainWindow` hooks `WM_GETMINMAXINFO` via `HwndSource.AddHook` to compute correct per-monitor bounds, and `StateChanged` toggles the root/title-bar/sidebar/content `Border`s' `Margin`/`CornerRadius` between the normal (rounded, 12px inset) and maximized (square, flush) look.

Text-box placeholder hints are rendered by the `ModernTextBox` control template in `Assets/Base.xaml` (a `TextBlock` bound to `Tag`, shown via a `Trigger` on `Text=""`) — set placeholder text through the `Tag` property on any `TextBox` using that style. There's no separate placeholder behavior class.

**Theming (`Assets/Base.xaml`/`Dark.xaml`/`Light.xaml`)**: what used to be a single `Themes.xaml` is now split — `Base.xaml` (typography, the `ProgressToScale` converter, and all control `Style`s) is merged statically in `App.xaml` and never changes; `Dark.xaml`/`Light.xaml` (the 15 `Color`/`Brush` keys plus 4 `*Subtle` badge brushes) are merged at runtime by `App.ApplyTheme(string theme)`, called from `OnStartup` right after `ISettingsService.LoadAsync()` and *before* `StartupUri` constructs `MainWindow` — swap the theme dictionary any later than that and the window would already be built against the wrong palette. The theme choice (`AppSettings.Theme`) only takes effect on next launch; there's no live-swap wiring, so don't expect `SettingsViewModel.Theme` to repaint the running UI.

**`UseWindowsForms=true`** (added for `NotifyIcon` balloon-tip notifications) collides with `UseWPF=true` under `ImplicitUsings=enable`: the SDK's generated global-usings file pulls in both `System.Windows` and `System.Windows.Forms`, and several common type names exist in both (`Application`, `UserControl`, `Brush`/`Brushes`/`Color`, `DragEventArgs`, `DataFormats`, `DragDropEffects`, `SaveFileDialog`). Any file that touches one of these needs an explicit `using X = System.Windows.X;` (or `Microsoft.Win32.X;` for `SaveFileDialog`) alias at the top — see `App.xaml.cs`, `Converters/Converters.cs`, `Views/DownloadView.xaml.cs`, `Views/HistoryView.xaml.cs`, `Views/SettingsView.xaml.cs`, `ViewModels/HistoryViewModel.cs` for the pattern. If a new ambiguous-type build error (`CS0104`) shows up after adding code that references a `System.Windows.*` type, this is almost always why — add the alias rather than fully qualifying every call site.

**`NotificationService`** (`Services/NotificationService.cs`) disposes its `NotifyIcon` on a `Task.Delay(6000)` timer rather than the `BalloonTipClosed` event — this is a WPF app with no `System.Windows.Forms.Application.Run()` message loop backing it, so WinForms balloon-tip lifecycle events aren't guaranteed to fire reliably.

**Playlist items intentionally leave `DownloadItem.FormatId` empty**: `DownloadViewModel.AddToQueue` enqueues one item per `VideoInfo.PlaylistEntries` entry (populated by `YtDlpService.FetchPlaylistAsync`'s `--flat-playlist --dump-single-json`, which returns no per-video format list) with `FormatId = string.Empty` on purpose — `BuildDownloadArgs` already falls back to `-f "{settings.DefaultQuality}"` whenever `FormatId` is blank, so don't "fix" this by trying to resolve formats per playlist entry.

All persisted data lives under `%LocalAppData%\YtDownloader\` (`settings.json`, `history.json`, `bin\yt-dlp.exe`, `bin\ffmpeg.exe`, `bin\ffprobe.exe`) — deleting that folder resets the app to a fresh-install state (yt-dlp ~19MB, ffmpeg ~290MB will re-download on next launch).
